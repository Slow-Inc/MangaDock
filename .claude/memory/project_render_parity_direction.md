User เทียบผลแปลของเรากับ meangrinch/MangaTranslator (หน้า benchmark One Punch-Man `ver:752fc515...`, JA→EN) แล้วตัดสินทิศทางปิด gap (2026-06-08):

1. **Line-break = narrow-column เหมือนเขา** — เลิก wrap ตาม bbox กว้างของ balloon, ไป wrap ตาม **safe interior ที่วัดจาก mask จริง** (distanceTransform + ray-cast, แบบ `calculate_centroid_expansion_box`) ให้อังกฤษ reflow เป็นคอลัมน์แคบตามรอยต้นฉบับญี่ปุ่น = ดูเป็นมังงะ **นี่คือรากของปัญหา "ดูเป็นย่อหน้า"** ไม่ใช่ขนาดฟอนต์
2. **Supersampling 4× เหมือนเขา** — render ที่ความละเอียดสูงแล้ว downscale (เขา max_font 16px + 4× → perceived size คุมได้, ขอบนุ่ม). เราตอนนี้ 1× + cap `h×0.5`→40-50px = ดูยักษ์ ทิศทาง: เพิ่ม supersampling ไม่ใช่แค่ cap 24
3. **Vertical แบบมังงะจริง** — ไม่ใช่ "คอลัมน์แคบแนวนอน" เฉยๆ แต่ stack ตัวอักษรแนวตั้งจริง (`_build_vertical_layout`) เมื่อ region สูงแคบ (aspect≥1.6, คำสั้น)
4. **SFX detector (#168) เอา** — AnimeText YOLO (`deepghs/AnimeText_yolo` ~400MB) opt-in ได้ เพราะ **VRAM เหลือ: เครื่อง dev ใช้ 5GB สูงสุด 7GB จาก 12GB** (มี headroom พอโหลด model ที่ 2)

**Why:** เห็น MangaTranslator คือ "as close to a human translator as possible" ([[project_translation_northstar]]) จริง — narrow-column + supersampling + vertical คือสิ่งที่ทำให้ของเขาดูเป็นมังงะ ของเราดูเป็น novel

**How to apply:** key insight = "wrap ตามอะไร สำคัญกว่าขนาดฟอนต์". ลำดับ: cheap wins ก่อน (cap size, center padding, wire #168 SFX) แล้วตามด้วยตัวโครงสร้าง (mask-aware narrow wrap + supersampling + vertical). มี doc ชำแหละเต็มกำลังทำที่ `docs/research/` (workflow translator-deep-dissection). เชื่อมกับ [[project_mangatranslator_study]]

**UPDATE 2026-06-13 — เจอรากจริงของ "narration ใหญ่/ล้น" = ตัว warp เอง → แก้ด้วย render-layout rework (PR #263, branch `feat/mit-clean-text-layout`):**
`render()` วาด text ลง canvas แล้ว **homography-warp เข้า detection quad เดิม**; narration ญี่ปุ่น quad เป็น **แนวตั้งสูงแคบ** → อังกฤษแนวนอนถูกยืด → ใหญ่/ล้น/ตัดบรรทัดเพี้ยน **ไม่มี knob ไหน (font cap #261, area-fit #166, ratio) แก้ได้สะอาด เพราะความเพี้ยน=ตัว warp**. ทางแก้ที่ถูก = `RenderConfig.clean_layout` (`MIT_CLEAN_LAYOUT`): สำหรับ region ที่ไม่ใช่บับเบิล/ไม่ใช่ SFX → เลิก warp, **วาง block แนวนอนตั้งตรงที่ font absolute เล็ก (`centered_box` + `clean_wrap_width`) ตรง center** → homography เป็น scale ล้วน. ยืนยันแล้วทั้ง direct-render (`MIT/tools/ab_clean.py` montage) + full-stack E2E ผ่าน Reader (narration/dialogue เล็ก/wrap/อยู่ใน panel ≈ reference). SFX ยังใหญ่ (path เดิม #168 ยกเว้น). **Winning config (`.env`):** `MIT_CLEAN_LAYOUT=1` + `MIT_FONT_SIZE_MAX=20` (= clean absolute font, ไม่ backfire เพราะ region `continue` ก่อนถึง legacy final_scale) + `MIT_ANTI_OVERLAP=1` + `MIT_EN_FONT=anime_ace_3.ttf` + `MIT_SUPERSAMPLING=4` + `MIT_BUBBLE_AREA_FIT` off + `MIT_SFX_DETECTOR=1` + `MIT_OCR_VLM_RESCUE=1`. Gotcha: `calc_horizontal` ต้องการ `max_height` เป็น positional. เชื่อมกับ [[project_benchmark_e2e_flow]]

**UPDATE 2026-06-13 (PR #264) — line-break ต้อง wrap ตาม footprint ต้นฉบับ:** `clean_wrap_width` ต้อง wrap ตาม **`bbox_w` (ความกว้างจริงของ region)** ไม่ใช่ `max(bbox_w,bbox_h)` — ไม่งั้น narration คอลัมน์สูงกลายเป็น "ย่อหน้ากว้างแบบ novel" (user ติ "ขึ้นบรรทัดไม่อ้างอิงต้นฉบับ"). wrap ตาม bbox_w → อังกฤษตัดบรรทัดตรงที่คอลัมน์ JP ตัด (แคบสูง + hyphenation อัตโนมัติจาก calc_horizontal) เหมือน target. floor 11% ของหน้าให้ dialogue สั้นได้ ~2 คำ/บรรทัด. **ห้ามใช้ความกว้าง bubble_box เป็น ref** — narration ก็ได้ bubble_box กว้างจาก seg → กว้างกลับ. **Cache gotcha:** wrap เป็น MIT code ไม่ขยับ `renderConfigHash` (อิง env เท่านั้น) → ต้อง `cache:reset`+restart backend ก่อน E2E ใน Reader ไม่งั้น serve patch เก่า

**UPDATE 2026-06-13 (PR #265) — ลบ text ไม่เนียน (gray blob ทับผมดำ) = patch crop เล็ก ขาด context ไม่ใช่ mask:** patch-mode inpaint crop ต่อ region → LaMa FFC global branch ขาด context ทั้งหน้า → hole ใหญ่ (text ใหญ่ทับ art ซับซ้อน/ดำ) เติม**เทา**. upstream `manga-image-translator-Original` + full-page path ของเราเอง inpaint **ทั้งหน้าครั้งเดียว** → เนียน. **วิธี diagnose ที่ชี้ขาด:** เทียบ `/translate/with-form/image` (full-page) vs `/translate/with-form/patches` (patch) — ถ้า full-page เนียนด้วย **mask เดียวกัน** = ปัญหา context ไม่ใช่ mask (CRF/pydensecrf เรามีครบ เหมือน upstream). **แก้:** `InpainterConfig.full_page_inpaint` (`MIT_PATCH_FULLPAGE_INPAINT`) — `translate_patches` inpaint ทั้งหน้า 1 ครั้ง แล้ว `PatchRenderer(full_inpainted=...)` slice background ให้ทุก patch, skip per-crop mask+inpaint. เร็วกว่า N per-group inpaint ด้วย. `tools/ab_fullpage.py` = harness เทียบ full-page vs patch

**UPDATE 2026-06-13 (PR #266) — "ทาสีทับ ไม่ใช่ลบ" = compositing (patch สี่เหลี่ยม) ไม่ใช่ inpaint:** หลัง full-page inpaint ยังเห็น**กล่อง/แถบจาง**หลัง text บนผมดำ เพราะ patch แปะเป็น**สี่เหลี่ยมทึบ** → ผม-LaMa (regenerate) แทน original ทั้งกรอบ แก้: **content-shaped patch alpha** (`patch_geometry.content_patch_alpha`) — opaque เฉพาะ (erase_mask ของ text เดิม ∪ stroke text ใหม่ที่ render ต่างจาก background), ที่เหลือโปร่งใส → original ทะลุมา ไม่มีกล่อง. gate `RenderConfig.content_patch` (`MIT_PATCH_CONTENT_ALPHA`), VRAM-free. **diagnose:** A/B controlled (full-page inpaint+layout เหมือนกัน ต่างแค่ content_patch) → rectangle เห็นกล่อง, content เห็นผมต่อเนื่อง. **⚠️ UPDATE: #266 ถูก REVERT (#267)** — /debug-mantra + **วัด luminance** พบ content_patch **ไม่ได้แก้ band** (rectangle band 201.5 vs content 201.3 = ไม่ขยับ) เพราะ band อยู่ตรง **erase region** ที่ alpha ต้องเก็บ opaque (ลบ text เดิม) → ยัง composite ผม-LaMa สว่างผิดตรงนั้น content_patch ลบได้แค่ margin ที่ไม่เคยมี band + ทำ text จาง. **บทเรียน: verify ด้วยการวัดตัวเลข ไม่ใช่ดู screenshot (เทียบคนละ translate run หลอกตา)**. **Flux ไม่คุ้มบนเครื่องนี้:** MIT ใช้ ~6GB/12GB → Flux Klein 4b FP8 (เล็กสุด) ~5-6GB → รวม ~11-12GB คาบเส้น OOM; LaMa polish (full-page+content-patch) ได้ผลพอโดยไม่ต้องเปลี่ยนโมเดล

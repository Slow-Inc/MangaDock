# SAFE Decomposition Map — MangaTranslator God Object (#187) + Model Lifecycle Abstractions (#188)

## 1. Overview

`MangaTranslator` (`MIT/manga_translator/manga_translator.py`, 3040 lines) is the single class through which every translation request flows, and it tangles at least a dozen orthogonal concerns — config parsing, global side-effects, stage orchestration, cross-page translation memory, model-unload TTL, debug-folder path lifecycle, and four near-duplicate post-translation pipelines — behind one untyped `Context` dict-bag and ~25 instance fields read deep inside methods. It is hard to extend because the same logic exists in **three-to-four subtly divergent copies** (single `_translate`, batch `_batch_translate_contexts`, concurrent `_concurrent_translate_contexts`, patch `translate_patches`), so any feature added to one path silently diverges from the others, and the differences (e.g. `min_ratio` 0.5 vs 0.3, region threshold `>=6` vs `>10`) are load-bearing, not bugs. The compounding risk is that `mode/share.py` holds a **process-lifetime singleton** of this object serving every web request serially, so its mutable lists (`all_page_translations`, `_original_page_texts`) bleed across unrelated jobs (#136/#140) and the only scrub (`reset_page_context`, line 2020) is called from `translate_patches` (2039) but **not** from `translate`/`translate_batch`. Each new model, translator, or render feature is bolted onto this core, multiplying the divergence — exactly the "becomes like LINE" failure the roadmap is written to prevent. The fix is not a rewrite but a sequence of **byte-identical extractions behind characterization tests**, each shippable, lowest-coupling first.

## 2. Responsibility map

Distinct responsibilities currently fused in the god object:

- **Construction & config parsing** — `parse_init_params` (L118-321) maps a flat dict to ~20 fields, validates device + `batch_concurrent>=batch_size>=2`, and mutates **process globals** (`ModelWrapper._MODEL_DIR`, `torch.backends.cuda.matmul.allow_tf32`).
- **Global logging side-effects** — `_setup_log_file` monkeypatches `builtins.print` and installs a FileHandler on every `__init__` (stacks on repeated construction).
- **Stage execution** — eight `_run_*` adapters (colorizer, upscale, detection, ocr, textline_merge, translation, inpainting, rendering), each already shaped `read ctx-subset → dispatch_* → return value`, but each also stamps `_model_usage_timestamps` and the translation stage reads cross-page memory.
- **Pipeline orchestration** — three+ drivers sequencing those stages: `_translate` (full, L442-632), `_translate_until_translation` (pre-half, L1649-1792), `_complete_translation_pipeline` (post-half, L2824-2917), and `translate_patches` (per-region-group, L2032-2218).
- **Mode dispatch** — `translate_batch` (L1446) forks individual / `_batch_translate_contexts` / `_concurrent_translate_contexts` plus a `psutil`/`gc`/`MemoryError` fallback ladder.
- **Text-translation dispatch** — the ChatGPT/ChatGPT2Stage-vs-`dispatch_translation` switch, duplicated in `_dispatch_with_context` (L1035-1094) and `_batch_translate_texts` (L2629-2758).
- **Cross-page translation memory** — `all_page_translations`/`_original_page_texts` + `_build_prev_context` (L958-1033) + `reset_page_context` (L2020).
- **Post-translation processing** — punctuation + post-dict + repetition-retry + target-lang-ratio retry + region filter, in **4 copies** (single L1162-1316, `_apply_post_translation_processing` L2760-2822, batch L2371-2401, concurrent L2542-2571; patch path applies almost none).
- **Model lifecycle / TTL eviction** — `_model_usage_timestamps`, `_unload_model` (L734-750), `_detector_cleanup_job` (L753-763), eager-preload (L412-422 ≡ L1672-1681).
- **Debug-folder path lifecycle** — `_set/_save/_restore_image_context`, `_get_image_subfolder`, `_result_path`, the `_current_image_context` swap closures (L2697-2732).
- **Verbose debug-image sink + web-streaming substitution** — scattered `if self.verbose: cv2.imwrite(...)` and the `_is_streaming_mode` 1×1 placeholder (L656).
- **Progress reporting** — `_report_progress` → mutable `_progress_hooks` (appended per-request by `share.py`).

## 3. Coupling and state

**How stages are coupled.** Every stage shares one untyped `Context` (`utils/generic.py:28`), a `dict` subclass whose `__getattr__` returns `self.get(item)` — so **any unset key reads as `None`, never raises**. Stages read/write arbitrary `ctx.*` attributes, and `_run_colorizer` even splats `**ctx` into `dispatch_colorization` (L687), forcing `Context` to stay Mapping-like and blocking a clean dataclass. Beyond `ctx`, stages reach back into ~25 instance fields (`device`, `verbose`, `context_size`, `batch_concurrent`, `_current_image_context`, …) and into process state: `os.environ['MANGA_OCR_RESULT_DIR']` (set/restored around `dispatch_ocr`, L786-797), `os.environ['PATCH_CONCURRENCY']` (L2095), and `_model_usage_timestamps` written as a side-effect of doing inference (8 sites).

## 5. Safe order

Every step is independently shippable and proven **byte-identical** by its test gate before the next begins (characterization net first, per the roadmap's Iron Rule 1). #188's model-lifecycle work (S3-S4, S20-S22) is **interleaved early** — not deferred to "Phase C, last" as the current plan has it — because the lifecycle seams are low-coupling, sit beside the stage code, and S15 (Stage protocol) needs S3 to keep stamping timestamps. The genuinely-hard #187 facade work (S23-S26) is last and depends on almost everything.

## 6. Roadmap reconciliation

**Does the current order hold?** Its *strategic* spine holds — "foundation → render → biggest-last, ship/validate between steps, characterization net first, core decomposition (#188→#187) is the objective." But its *tactical* claim that #188 and #187 are two monolithic Phase-C items done strictly after all of Phase A/B is **wrong in practice and already contradicted by reality**: #187 is not one step, it is ~16 independently-shippable seams (S1, S2, S6-S11 are low-risk and have **nothing** to do with line-break/render/config), and several have **already been landed out of order** — `correct_punctuation` (`punctuation.py`), `check_repetition_hallucination`/`check_target_language_ratio` (`translation_checks.py`), the `_greedy_pack` line-break seam (#186), and #192's partial config slices. So the plan's own "Status" (#187–#193 "open") is stale.

**KEEP:**
- The anti-compounding principle and the four Iron Rules (characterization-first, report-on-close, ship-between-increments, no-merge-without-confirm) — apply verbatim to every seam above.
- Phase A foundation items as genuine prerequisites: **#192 (config centralize)** unblocks **S12 PipelineParams** and de-risks construction globally; **#193 (worker lifecycle)** is cheap and independent.
- **#191 investigate-first** (dead-LDM/YOLOv5 deletion) and the render seams **#189/#190** — these are orthogonal to the god-object decomposition and can run in parallel; keep them where they are.
- **#188 before #187** as a *facade* dependency: S23 StageRunner / S15 Stage protocol genuinely benefit from S3/S4 existing first.

**REORDER:**
- **Split #187 and #188 into their constituent seams and interleave them**, do not treat either as one big-bang Phase-C PR.

01 — ARCHITECTURE

สถาปัตยกรรมภายใน MIT

MIT รันเป็น 2 process เพื่อแยกงาน ML ที่หนักออกจาก HTTP layer (T4-STANDARD Pillar 4: Worker Memory Contract)

Backend
ส่ง page URLs ทั้งหมด + callback_url
POST /translate/with-form/patches/batch
HTTP (multipart)
MIT — Web Server · server/main.py · :5003
คิวงาน + executor pool + ส่ง webhook
run_batch_with_callbacks() · send_webhook() · myqueue.py
pickle over HTTP (localhost)
MIT — Worker · mode/share.py · :5004
โหลด ML models + แปลจริง
translate_patches() → คืน Patch Set (detect · ocr · translate · inpaint · render)

ทำไมต้องแยก 2 process?

Web Server ต้องตอบ HTTP เร็วและรับงานขนานได้ ส่วน Worker ถือ ML models (GPU) ที่กิน RAM/VRAM มหาศาล การแยกทำให้ web layer ไม่ตายตาม worker และ restart worker ได้โดยไม่กระทบ HTTP — แต่การสื่อสารใช้ pickle over HTTP ซึ่งเป็นที่มาของบั๊กความปลอดภัย #103

02 — TRANSLATION PIPELINE

Pipeline การแปล 1 หน้า

เมื่อ Worker รับรูป 1 หน้า มันวิ่งผ่าน stage ต่อไปนี้ใน translate_patches() แต่ละ stage มี try/except + ignore_errors กันล้ม

  1. Colorize / Upscale (ข้ามถ้าไม่เปิด) — เตรียมรูป
  2. Detection _run_detection — หา textline (กล่องข้อความ) + mask
  3. OCR _run_ocr — อ่านตัวอักษรในแต่ละกล่อง
  4. Textline Merge textline_merge.dispatch — รวมกล่องที่อยู่ใกล้กันเป็น region · บั๊ก #111 (prob)
  5. Text Translation _run_text_translation → dispatch → Gemini / Qwen3 · บั๊ก #107 / #108
  6. Post-translation Check _check_target_language_ratio — ตรวจคุณภาพ · บั๊ก #109
  7. Per-region: Mask → Inpaint → Render rendering.dispatch — ลบข้อความเดิม + วาดคำแปล · บั๊ก #110
  8. Encode PNG — ได้ Patch Set คืนกลับ Web Server
จุดสำคัญ

Pipeline แกนหลัก (translate_patches + patch helpers) เขียนดีและกันล้มครบ — บั๊กส่วนใหญ่อยู่ที่ translator layer (Gemini/sample cache) และ การส่งผลกลับ (webhook) ไม่ใช่ตัว ML

03 — WEBHOOK DELIVERY

เส้นทาง Webhook (โดเมนของ #100)

หลังแปลเสร็จแต่ละหน้า MIT ต้องส่ง Patch Set กลับ Backend ผ่าน webhook — นี่คือจุดที่ Patch Set “หาย” จนเกิดอาการ 0/20

  1. Frontend กด “แปลทั้งตอน” → POST /books/chapters/:id/batch-translate-patches (เปิด SSE)
  2. Backend register Batch Job → ส่ง page URLs + callback_url ไป MIT → MIT ตอบ 202 ทันที
  3. MIT run_batch_with_callbacks() วน per page เบื้องหลัง: แปล → send_webhook()
  4. Backend /webhooks/mit/callback → เซฟ patch + cache + notify SSE → Frontend วาด overlay

โค้ดปัจจุบันของ send_webhook() — จุดที่ #100 จะแก้

async def send_webhook(url, secret, payload):
    data = json.dumps(payload, ...).encode("utf-8")
    headers = {"Content-Type": "application/json"}
    if secret:
        headers["x-mit-signature"] = hmac_sign(secret, data)

    async with httpx.AsyncClient() as client:
        try:
            await client.post(url, content=data, headers=headers, timeout=60.0)  # ① ไม่เช็ค status
        except Exception as e:
            print(f"Webhook delivery failed to {url}: ...")  # ② fail = print แล้วทิ้ง ไม่ retry
① ไม่เช็ค HTTP status

Backend ตอบ 500 / 413 → MIT ถือว่า “สำเร็จ” เพราะ post() ไม่ throw บน non-2xx

② fail = print ไม่ retry

network blip / backend restart / 5xx ชั่วคราว → Patch Set หายทันที ไม่มี retry, ไม่มี dead-letter

ทำไมถึงเป็น “0/20”

GPU แปลครบทุกหน้าจริง แต่ผลส่งไม่ถึง Backend → SSE ไม่มีข้อมูล → Frontend ค้างที่ 0/20 และ ไม่มี reconciliation ที่จะดึงหน้าที่หายกลับมา (ตาม CONTEXT.md) — #100 retry คือแนวป้องกันเดียว

05 — BUG REPORT

บั๊กที่เจอ (#100 – #111)

จากการ scrutinize logic ทั้งหมด (ยกเว้น model AI) คลิกการ์ดเพื่อดูรายละเอียด · กรองตามความรุนแรงได้

Critical — กระทบอาการจริง Security Major — bug จริง Cleanup — dead code
#100
send_webhook ไม่ retry และกลืน error → Patch Set หายถาวร
server/main.py · send_webhook()
Critical

webhook ยิงครั้งเดียว ไม่เช็ค status ถ้า fail แค่ print แล้วทิ้ง

Root cause

non-2xx ไม่ throw + except แค่ print ไม่ retry ไม่มี dead-letter

Impact

คือสาเหตุของ “0/20” — GPU แปลครบ แต่ผลส่งไม่ถึง Backend หายถาวร (ไม่มี reconciliation)

#101
ยกเลิก Batch ไม่ propagate ไป MIT → zombie job เผา GPU
server/main.py · run_batch_with_callbacks · DummyRequest.is_disconnected
Critical

DummyRequest.is_disconnected() return False เสมอ → MIT ไม่มีทางรู้ว่า user ยกเลิก

Root cause

webhook path ใช้ stub request ที่บอกว่า “ยัง connect อยู่” ตลอด → ไม่ self-cancel

Impact

ยกเลิกแล้ว MIT แปลต่อครบ 20 หน้า เผา GPU; retry chapter เดิม → 2 background task ชนกัน = ที่มาของ “cancel แล้วแย่ลง”

#102
Path traversal + endpoint จัดการไฟล์ไม่มี auth
server/main.py · /result/{folder} · /results/{folder} · /results/clear
Security

Root cause

ไม่ validate folder name; endpoint เปิด unauthenticated บน 0.0.0.0

Impact

อ่าน / ลบ directory นอก RESULT_ROOT ได้ (มี shutil.rmtree)

06 — DESIGN GRILL

Grill: ออกแบบการแก้ #100 (webhook retry)

คำถามออกแบบที่ต้องตัดสินใจก่อน TDD — ไล่ลง design tree ทีละข้อ แต่ละข้อมี คำแนะนำของผม highlighted คลิก choice เพื่อ mark ความคิดคุณ (ไม่ผูกพัน — ตอบจริงใน chat)

บริบทจาก Roadmap.md + CONTEXT.md (กำหนด recommendation)

Pillar 1 Idempotent: Backend dedupe ด้วย pageIndex แล้ว → retry/re-send ปลอดภัย · Pillar 3: convention คือ retry budget MAX_RETRIES=5 + dead-letter · Pillar 6: structured JSON log · ไม่มี reconciliation → #100 retry คือ defense เดียว

QUESTION 1 / 6 — Retryable failure classification
status ไหนควร retry?
Acceptance criteria เดิมเขียนว่า retry “5xx/4xx” — แต่ 4xx ส่วนใหญ่ retry ไปก็ผลเดิม
QUESTION 2 / 6 — Retry count + backoff
กี่ครั้ง และ backoff แบบไหน?
โปรเจกต์มี convention MAX_RETRIES=5 (cache write-behind)
QUESTION 3 / 6 — Per-attempt timeout
timeout ต่อครั้งเท่าไหร่?
ปัจจุบัน timeout=60.0 — ถ้า retry หลายครั้งจะรอนานมาก
QUESTION 4 / 6 — Blocking vs decoupled
retry ควร block หน้าถัดไปไหม?
ปัจจุบัน loop await send_webhook แบบ sequential ต่อหน้า
QUESTION 5 / 6 — Dead-letter mechanism
หมด retry แล้วทำอะไรกับ Patch Set ที่หาย?
MIT ไม่มี Redis (dead-letter queue อยู่ฝั่ง Backend cache layer)
QUESTION 6 / 6 — Configurability
retry bounds ผ่าน env var?
#100 acceptance criteria ระบุ “configurable via env var with sensible defaults”
หลังตอบครบ → ขั้นต่อไป

สรุป design ที่ตกลง → เขียน failing test (TDD RED) สำหรับ send_webhook: mock httpx ให้คืน 500 แล้ว 200 → assert ว่า retry แล้วสำเร็จ + assert dead-letter log บน 4xx → จากนั้นเขียน implementation ให้ผ่าน (GREEN) · คำถาม blocking: MIT มี pytest setup สำหรับ server หรือยัง? (จะเช็คให้ก่อนเขียน test)
